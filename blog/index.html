Architecture Deep Dive

Signals in,
HTML out.
Rust in between.

How FormaJS compiles reactive components to a binary IR, renders them in Rust, and hydrates with fine-grained signals — no virtual DOM, no framework runtime, no compromise.

01 — The Problem

Every framework makes you choose

Simple but limited — Alpine.js, htmx, Stimulus. You get progressive enhancement and server-rendered HTML, but the moment you need a reactive data table or an optimistic UI update, you're reaching for a second tool.

Powerful but complex — React, Vue, Svelte. You get the full component model, but now you need a build step, a bundler, a meta-framework, and your server-rendered HTML is a hydration away from being useful.

We wanted one reactive core that scales from a <script> tag to a full-stack Rust SSR pipeline. A library where the same signal graph powers a data-on:click directive and a compiled JSX component. Where the server can render your components without touching Node.js.

That's FormaJS.

What's FMIR?

Forma Module IR is a compact binary format — think bytecode for your UI. The compiler analyzes your h() calls and emits a stream of opcodes: OPEN_TAG, DYN_TEXT, SHOW_IF, LIST, ISLAND_START, and more. Static content gets baked in. Dynamic values become slots — typed placeholders the server fills at render time. 

// Binary layout (little-endian)
[Header: 16 bytes]
  Magic: "FMIR"    Version: 2    Flags: 0    Source hash: u64

[Section Table: 32 bytes]
  Bytecode   ← opcode stream (OPEN_TAG, DYN_TEXT, ISLAND_START...)
  Strings    ← tag names, attribute keys, text content
  Slots      ← typed signal metadata with SSR defaults
  Islands    ← component boundaries for client hydration

[Data Sections]
  ← compact, sequential, zero-copy parseable

The Rust walker

The forma-ir crate reads the FMIR binary and walks the opcode stream linearly, emitting HTML as it goes. When it hits a DYN_TEXT opcode, it looks up the slot value (provided by the server as JSON) and wraps the output in marker comments that FormaJS uses later for hydration: 

<!-- The Rust walker emits this HTML -->
<div class="dashboard">
  <h1>
    <!--f:t0-->Count: 0<!--/f:t0-->
  </h1>
  <button
    data-forma-island="0"
    data-forma-component="Counter"
    data-forma-status="pending"
    data-forma-hydrate="load">
    +1
  </button>
</div>

Those <!--f:t0--> comments are the protocol. The Rust walker writes them. FormaJS reads them. They're how a signal update in the browser knows exactly which text node to mutate — no diffing, no tree walk, no reconciliation overhead.

03 — Three Ways In

One signal engine, three entry points

FormaJS gives you three ways to build UI, all sharing the same reactive graph. Pick the right tool for the job. Upgrade without rewriting.

Zero build step

HTML Runtime

Drop a <script> tag, write data-* attributes. Alpine.js-style progressive enhancement with a CSP-safe expression parser. No bundler, no config, no tooling.

Hyperscript

h() Function

Preact-style h('div', {onClick}, 'hello'). Returns real DOM elements, not virtual nodes. Fine-grained signals update only what changed.

JSX

Full Component Model

Same h() function with JSX syntax. Configure jsxFactory: "h" in your bundler and write familiar component code.

HTML Runtime: the Alpine.js-inspired path

<script src="https://unpkg.com/@getforma/core/dist/formajs-runtime.global.js"></script>

<div data-forma-state='{"count": 0}'>
  <p data-text="{count}"></p>
  <button data-on:click="{count++}">+1</button>
</div>
Design Decision

We use data-on:click instead of Alpine's @click shorthand. It's longer, but it's valid HTML5, unambiguous for AI code generation (our pipeline generates these directives programmatically), and parseable by any standard HTML parser. The data-* namespace is a feature, not a limitation.

Hyperscript: the Solid.js-inspired path

import { createSignal, h, mount } from '@getforma/core';

const [count, setCount] = createSignal(0);

mount(() =>
  h('button', { onClick: () => setCount(count() + 1) },
    () => `Clicked ${count()} times`
  ),
  '#app'
);

Notice: h('button') returns a real HTMLButtonElement. Not a virtual node. Not a proxy. A real DOM element. The signal count updates only the text node inside the button — nothing else re-renders.

04 — Reactivity

Fine-grained signals, real DOM

FormaJS's reactive core is inspired by SolidJS and backed by alien-signals. When a signal changes, only the specific DOM text node or attribute that depends on it updates — not the whole component tree.

Why alien-signals?

We started with Preact Signals. It worked. But alien-signals gives us pauseTracking/resumeTracking for critical sections during SSR and hydration, tracks the TC39 Signals proposal more closely, and has no framework coupling. We import exactly 7 public API symbols. No internals. 

import { createSignal, createEffect, createComputed, batch } from '@getforma/core';

const [count, setCount] = createSignal(0);
const doubled = createComputed(() => count() * 2);

createEffect(() => console.log('doubled:', doubled()));

batch(() => {
  setCount(1);
  setCount(2); // effect fires once with value 4
});

The API is intentionally familiar to Solid users. createSignal returns a [getter, setter] tuple. createEffect auto-tracks dependencies. batch defers notifications. If you know Solid, you know FormaJS.

05 — Islands

Hydrate what matters, skip the rest

Inspired by Astro's Islands Architecture, FormaJS hydrates independent regions of server-rendered HTML. Each island is self-contained, error-isolated, and only loads JavaScript when its trigger fires. 

import { activateIslands, createSignal, h } from '@getforma/core';

activateIslands({
  Counter: (el, props) => {
    const [count, setCount] = createSignal(props?.initial ?? 0);

    // el is the island's root HTMLElement
    el.classList.add('is-hydrated');

    return h('div', null,
      h('span', null, () => String(count())),
      h('button', { onClick: () => setCount(c => c + 1) }, '+1'),
    );
  },
});

The Rust IR walker defines four hydration triggers — matching Qwik's insight that not all interactivity needs to load immediately:

Trigger When it hydrates Use case
load Immediately on page load Above-the-fold interactive content
visible When island enters viewport Below-the-fold components
idle During browser idle time Non-critical functionality
interaction On first user interaction Skeleton+skin pattern

Descriptor-based hydration

When FormaJS hydrates an island, it doesn't re-render the DOM. Instead, it runs your component in hydration mode where h() returns plain descriptor objects instead of DOM elements. Then a top-down walk (adoptNode) matches these descriptors against the existing SSR DOM, attaching event handlers and reactive bindings without creating a single new element.

The synchronization protocol between Rust and JavaScript is a set of HTML comment markers:

Marker Written by Read by Purpose
<!--f:t0--> Rust walker FormaJS hydrate.ts Dynamic text node boundary
<!--f:s0--> Rust walker FormaJS hydrate.ts Show/if conditional boundary
<!--f:l0--> Rust walker FormaJS hydrate.ts List rendering boundary
<!--f:i0--> Rust walker FormaJS activate.ts Island boundary

Cleanup: disposing islands on module swap

In Forma's AI pipeline, modules get swapped constantly inside <forma-stage> Shadow DOM during generation. Each island creates a createRoot scope with effects, listeners, and signal subscriptions. Without cleanup, every swap leaks. 

import { deactivateAllIslands } from '@getforma/core';

// Before swapping module content in a shadow root:
deactivateAllIslands(shadowRoot);
shadowRoot.innerHTML = newModuleHtml;

// Or dispose a single island:
import { deactivateIsland } from '@getforma/core';
deactivateIsland(islandElement);
06 — Security

CSP-safe by design

The HTML Runtime includes a hand-written expression parser that handles identifiers, dot access, optional chaining, ternaries, template literals, method calls, array literals, and more — all without eval() or new Function().

The standard build uses new Function() as a fallback for complex expressions. The hardened build locks that off entirely at compile time — the flag __FORMA_UNSAFE_EVAL_MODE__ is set to "locked-off" by the bundler, making it impossible to toggle at runtime. 

<!-- Strict CSP: no eval, no Function, ever -->
<meta http-equiv="Content-Security-Policy"
      content="script-src 'nonce-abc123' 'self'">

<script nonce="abc123"
  src="formajs-runtime-hardened.global.js"></script>

<!-- This just works. No CSP violations. -->
<div data-forma-state='{"items": []}'>
  <p data-text="{items.length > 0 ? items.length + ' items' : 'Empty'}"></p>
</div>

The Rust server generates CSP headers with nonce injection for every request. Every inline <script> and <style> gets a unique nonce. No 'unsafe-inline', no 'unsafe-eval'.

07 — Proof

Proven in a production auth system

The full Forma stack was proven in GateWASM — a multi-tenant authentication platform with OAuth 2.0 + PKCE, EdDSA JWT signing, token rotation with family tracking, brute-force protection, and HIPAA compliance mode. 256 tests. Deployed to AWS on App Runner + RDS PostgreSQL.

08 — Inspirations

Standing on the shoulders of giants

Every design decision in FormaJS was informed by studying what the best frameworks got right — and where they made trade-offs we wanted to avoid.

SolidJS

Fine-grained reactivity

Signals + real DOM is the correct performance trade-off. We adopted Solid's [getter, setter] tuple API, effect auto-tracking, and the principle that components run once (not on every render).

Alpine.js

HTML-first progressive enhancement

The data-* directive model — declare state in HTML, bind with attributes, no build step. We built a CSP-safe expression parser to go further than Alpine's eval()-based approach.

Qwik

Deferred hydration triggers

Not all interactivity needs to load immediately. Our four hydration triggers (load, visible, idle, interaction) let you ship less JavaScript upfront.

Astro

Islands architecture

Independent interactive regions within server-rendered HTML. Each island has its own createRoot scope with error isolation — a broken island never crashes its siblings.

Hotwire / Turbo

Server-driven UI

The server should be able to update the UI without the client re-rendering everything. Our reconciler with PRESERVE/RESET/REPLACE scope modes, plus SSE/WebSocket primitives, enables Turbo Frames-style partial replacement.

Lit

Custom elements & Shadow DOM

Root element access during hydration is essential. Our (el, props) callback signature mirrors the pattern every Web Component framework uses — this is the element.

// main.ts
import { createSignal, h, mount } from '@getforma/core';

const [name, setName] = createSignal('World');

mount(() =>
  h('div', null,
    h('h1', null, () => `Hello, ${name()}!`),
    h('input', {
      value: name,
      onInput: (e) => setName(e.target.value),
    }),
  ),
  '#app'
);

That's it. No virtual DOM to understand. No hooks rules to memorize. No compiler to configure. Signals update the DOM. The DOM is the source of truth.
